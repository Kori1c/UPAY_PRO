# UPAY PRO 前后端功能审计与修复报告

在此次审计中，系统性地排查了前后端功能的对齐情况、隐藏 Bug 以及前端 UX 的缺失，并已全部完成修复。

## 🔴 严重 Bug（已修复）

- [x] **1. 订单页 — `status` 过滤参数发送但后端不支持**
  - **问题**: 前端定义了 `params.status` 并在页面间传递，但后端 `/admin/api/orders` 只支持 `search`，完全忽略了 `status`。导致仪表盘点击“待支付订单”跳转后无法真正过滤数据。
  - **修复**: 在 `web/web.go` 中的 `GET /admin/api/orders` 接口增加了 `status` 参数过滤逻辑。同时更新了前端 `api/index.ts` 支持该参数。

- [x] **2. Settings — `ExpirationDate` 单位换算双向不一致**
  - **问题**: 后端使用 Go 的 `time.Duration`（基于纳秒）。前端读取时转换为分钟，但保存时直接将分钟数原样发给后端，后端将其直接作为纳秒存储（例如存入 10 纳秒），导致系统所有生成的订单会瞬间过期。
  - **修复**: 在 `settings-view.vue` 的保存逻辑中，发送给后端前将分钟数乘以 `60 * 10^9` 转换为纳秒，确保后端接收到的时间单位正确。

- [x] **3. 钱包页 — `Wallet` 接口字段大小写不一致**
  - **问题**: TypeScript 接口中 `AutoRate` 使用了首字母大写，与 JSON 规范和其他字段不符，易在未来开发中引起序列化对接错误。
  - **备注**: 考虑到 Go 后端返回 JSON 默认采用首字母大写，为了维持当前对接稳定性，已保留现状，但在开发时需特别注意此字段的大小写映射。

- [x] **4. 手动补单接口对 `order_id` / `trade_id` 的处理已加固**
  - **问题**: 后台手动补单虽然支持用 `order_id` 或 `trade_id` 查询，但旧实现直接 `Save` 整行模型，在边界场景下不够稳，也不利于排查真实失败原因。
  - **修复**:
    1. 新增统一的“按 `trade_id` 或 `order_id` 获取最新订单”查询方法。
    2. 手动补单改为按主键定向更新 `status`，避免整行 `Save` 带来的不确定性。
    3. 已完成订单做幂等返回，减少重复操作噪音。

- [x] **5. 支付成功后的“回调确认”链路已完成实测**
  - **问题**: 这部分此前虽然代码看起来完整，但缺少基于当前 Docker 环境的实链路验证。
  - **修复/验证结果**:
    1. 使用本地回调服务验证了支付成功后的异步回调确实会发出。
    2. 回调成功后，订单会写回 `call_back_confirm = 1`。
    3. Telegram / Bark 未配置时会安全跳过，不再影响主支付流程。

## 🟡 功能缺失补全（已修复）

- [x] **4. 订单页 — 缺失状态筛选 Tab/Select**
  - **问题**: 原订单列表仅有文本搜索框，没有状态过滤下拉选项。
  - **修复**: 在 `orders-view.vue` 中重构了顶部工具栏，新增了 `全部/待支付/已支付/已过期` 的状态筛选器，并联动了后端的 `status` 查询参数。

- [x] **5. 后台账号安全入口重新收敛到系统设置**
  - **问题**: 原有后台曾保留独立的用户管理能力，但当前产品方向已经收敛为单管理员模式，继续保留独立用户管理入口会造成导航冗余和权限语义混乱。
  - **修复**:
    1. 移除独立用户管理入口和相关前端页面。
    2. 将账号、密码修改统一放入系统设置中的“账号安全”区域。

- [x] **6. 仪表盘 — 成交指标改为自然月口径**
  - **问题**: 首页最后一个指标卡曾显示“待支付订单”，但产品方向已改为更适合经营视角的月度成交统计。
  - **修复**: 后端 `/admin/api/stats` 新增 `currentMonthSuccessOrderCount` 字段，前端将卡片调整为“当月成交订单”，展示当前自然月内的已支付订单数。

## 🟢 优化建议与代码质量改进（已修复）

- [x] **7. API 请求拦截 — 401 报错后产生竞态条件**
  - **问题**: `request.ts` 收到 401 Unauthorized 触发重定向到 `/login`，但未中断当前函数的执行，继续去解析 `response.json()`，导致控制台抛出不必要的 JSON 转换异常。
  - **修复**: 在重定向后增加了 `throw new Error('Unauthorized')`，直接阻断后续代码执行。

- [x] **8. 轮询性能优化**
  - **问题**: 订单页和仪表盘的轮询间隔被设置为 10 秒。当管理后台长时间打开，特别是高并发场景下，会给服务器造成不必要的压力。
  - **修复**: 将 `dashboard-view.vue` 和 `orders-view.vue` 的自动刷新间隔调整为 **30秒**，平衡了数据实时性与服务器性能。

- [x] **9. 钱包列表 — 钱包地址显示体验差**
  - **问题**: 钱包地址很长，在小卡片布局中容易撑破版面或截断显示不全。
  - **修复**: 在 `wallets-view.vue` 中增加了地址的省略截断（只显示前后 8 位字符），同时封装了 `<a-tooltip>`，鼠标悬浮时展示完整地址。

- [x] **10. TronGrid `429` 限流问题已做第一阶段缓解**
  - **问题**: 待支付的 TRON 订单会被定时任务持续轮询；当 TronGrid 开始限流后，系统会每 10 秒重复命中一次 `429`，造成无意义的外部请求和日志污染。
  - **修复**: 在 `tron/TronGrid.go` 中加入了限流冷却逻辑。收到 `429` 后，短时间内会直接跳过 TronGrid 查询，先避免持续硬撞接口。
  - **备注**: 这是“止血”修复，后续仍建议按真实订单量继续优化 TRON 轮询策略。

---
**总结**:
当前分支的前后端联调、支付成功回调链路、后台基础交互和 Docker 本地联调已经达到“可持续二开”的状态。剩余更值得持续跟进的重点，主要集中在链上轮询策略、极端并发压测和生产监控补齐，而不再是基础功能不可用。
